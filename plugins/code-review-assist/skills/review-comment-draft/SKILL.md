# レビューコメント下書き

レビューで発見した指摘事項を、チームメンバーに伝わりやすい「理由 + 提案」形式のコメントとして整形する。
PR へのインラインコメントとしてそのまま使える形式で出力する。

## ツール使用方針

- ファイル読み取りは xcodeproj-mcp-server の利用を優先する
- MCP が利用できない場合は Read / Grep ツールで直接ファイルを読み取る
- PR へのコメント投稿は行わない（下書きのみ）

## 入力

以下のいずれかを受け取る:

- レビュー指摘の概要（自然言語）
- pr-diff-review や architecture-conformance の出力結果
- ファイルパスと行番号の指定

## コメント構成ルール

各コメントは以下の構成に従う:

### 1. 重要度ラベル

| ラベル | 用途 |
|---|---|
| `[Must Fix]` | マージ前に必ず修正が必要 |
| `[Should Fix]` | 強く推奨する修正 |
| `[Suggestion]` | より良い書き方の提案 |
| `[Question]` | 意図の確認（指摘ではない） |
| `[Nit]` | 些細な指摘（修正は任意） |

### 2. 指摘内容

- 何が問題かを具体的に説明する
- コードの該当箇所を引用する

### 3. 理由

- なぜ問題なのかを説明する
- バグ・パフォーマンス・保守性・規約等の観点を明示する

### 4. 改善提案

- 具体的なコード例を含む修正案を提示する
- Swift 6.2 準拠のコード例を使用する

## 出力

各コメントを以下の形式で出力する:

```
### <ファイル名>:<行番号>

**[<重要度ラベル>]** <指摘の要約>

<指摘内容の詳細>

```swift
// 現在のコード
<問題のあるコード>
```

**理由:** <なぜ問題なのかの説明>

**提案:**

```swift
// 改善案
<修正後のコード例>
```
```

## コメント作成のガイドライン

### 良いコメントの例

```
### UserViewModel.swift:15

**[Should Fix]** ObservableObject の使用

`ObservableObject` + `@Published` は Swift 6.2 では非推奨です。

```swift
// 現在のコード
class UserViewModel: ObservableObject {
    @Published var name: String = ""
}
```

**理由:** Observation フレームワーク（`@Observable`）はパフォーマンスが優れており、
プロパティ単位の変更追跡が可能です。チームの Swift 6.2 移行方針にも沿っています。

**提案:**

```swift
// 改善案
@Observable
class UserViewModel {
    var name: String = ""
}
```
```

### 避けるべきコメント

- 理由なしの指摘（「これは良くない」のみ）
- 代替案を示さない否定
- 主観的な好み（客観的根拠を伴わない「こう書くべき」）
- 同じ問題の繰り返し指摘（まとめて 1 つのコメントにする）
