# Issue トリアージ

指定された issue の内容を分析し、適切なラベル・優先度・担当者を提案する。

## ツール使用方針

- `gh` CLI を使用して issue 情報を取得・更新する
- リポジトリのラベル一覧・メンバー情報の取得にも `gh` CLI を使用する

## 実行フロー

### Step 1: issue 情報の取得

```bash
# issue の詳細を取得
gh issue view <issue-number>

# リポジトリのラベル一覧
gh label list --limit 100

# チームメンバー情報（collaborators）
gh api repos/{owner}/{repo}/collaborators --jq '.[].login'
```

### Step 2: 内容分析

issue のタイトルと本文から以下を判定する。

#### カテゴリ判定

| カテゴリ | 判定基準 |
|---|---|
| bug | 「不具合」「エラー」「クラッシュ」「修正」など |
| feature | 「新機能」「追加」「実装」「対応」など |
| improvement | 「改善」「リファクタリング」「パフォーマンス」など |
| documentation | 「ドキュメント」「README」「コメント」など |

#### 優先度判定

| 優先度 | 判定基準 |
|---|---|
| critical | クラッシュ、データ損失、セキュリティに関連 |
| high | ユーザー影響が大きい、ブロッカーとなっている |
| medium | 通常の機能追加・改善 |
| low | 軽微な修正、ドキュメント更新 |

#### 担当者提案

- 変更対象のモジュール・ファイルに対する過去のコミット頻度から適任者を推定する
- `gh api` でリポジトリの collaborators を取得し、候補を提示する

### Step 3: 提案の出力

分析結果をユーザーに提示する。ラベル付与やアサインの実行はユーザーの確認を得てから行う。

## 出力

```
## Issue トリアージ提案: #<issue-number>

### 現在の状態
- タイトル: <タイトル>
- ラベル: <現在のラベル>
- 担当者: <現在の担当者>

### 提案
- カテゴリ: <カテゴリ>
- 優先度: <優先度>
- 推奨ラベル: <ラベル1>, <ラベル2>
- 推奨担当者: <担当者> （理由: <理由>）

### 根拠
- <判定の根拠を説明>
```

## 対象の特定

- 引数で issue 番号が指定された場合はその issue を分析する
- 指定がない場合はラベルなしの open issue を一覧から選択する
