# PR 差分レビュー

PR の差分を取得・解析し、コード品質・設計・安全性の観点からレビューコメントを生成する。
レビュー観点の詳細は → **references/REVIEW_CRITERIA.md** を参照。

## ツール使用方針

- PR 差分の取得は `gh` CLI を使用する（`gh pr diff`）
- ファイルの詳細確認は xcodeproj-mcp-server の利用を優先する
- MCP が利用できない場合は Read / Grep ツールで直接ファイルを読み取る

## 入力

- PR 番号またはブランチ名が指定された場合はそれを使用する
- 指定がない場合はカレントブランチの差分（`git diff main...HEAD`）を対象にする

## 差分取得

```bash
# PR 番号が指定された場合
gh pr diff <PR番号>

# カレントブランチの場合
git diff main...HEAD
```

## レビュー観点

### 1. コード品質

- 不要なコード・デッドコードが含まれていないか
- マジックナンバー・ハードコードされた文字列がないか
- エラーハンドリングが適切か（`try?` の乱用、空の `catch` ブロック等）
- force unwrap（`!`）/ force cast（`as!`）の使用

### 2. Swift 6.2 準拠

- `@Observable` を使用しているか（`ObservableObject` / `@Published` は非推奨）
- `Sendable` 準拠・actor 分離が適切か
- 旧 API（`@StateObject`, `@ObservedObject`, `@EnvironmentObject`）が使われていないか
- `async/await` パターンが適切か

### 3. 設計・アーキテクチャ

- MVVM パターンの分離が守られているか
- レイヤー間の依存方向が正しいか
- DI（Dependency Injection）が適用されているか
- Protocol 指向設計の原則に従っているか

### 4. 安全性・パフォーマンス

- メモリリーク（循環参照）のリスクがないか
- メインスレッドでの重い処理がないか
- 不要な再描画を引き起こすコードがないか

### 5. テスタビリティ

- テスト可能な設計になっているか
- テストコードが追加されているか（新機能・バグ修正の場合）

## 出力

```
## PR 差分レビュー

### サマリー
- 変更ファイル数: N
- 追加行数: +N / 削除行数: -N
- レビュー指摘数: N（Critical: N, Warning: N, Info: N）

### 指摘一覧

#### [Critical] <ファイル名>:<行番号>
- 指摘内容: <具体的な問題の説明>
- 理由: <なぜ問題なのか>
- 提案: <改善案とコード例>

#### [Warning] <ファイル名>:<行番号>
- 指摘内容: <具体的な問題の説明>
- 理由: <なぜ問題なのか>
- 提案: <改善案とコード例>

#### [Info] <ファイル名>:<行番号>
- 指摘内容: <改善の余地がある箇所>
- 提案: <より良い書き方の提案>

### 良い点
- <差分の中で良い実装があれば言及する>
```
