# 構造化 Issue 作成

> このスキルは `/issue-create` で明示的に実行します。GitHub に issue を作成する副作用があるため、自動活性化されません。

対話的にヒアリングし、構造化された issue を GitHub に作成する。

## ツール使用方針

- `gh` CLI を使用して issue を作成する
- リポジトリのラベル一覧・テンプレート情報の取得にも `gh` CLI を使用する

## 実行フロー

### Step 1: リポジトリ情報の収集

リポジトリに設定されている情報を収集する。

```bash
# ラベル一覧の取得
gh label list --limit 100

# issue テンプレートの有無を確認
ls .github/ISSUE_TEMPLATE/ 2>/dev/null || echo "テンプレートなし"
```

### Step 2: issue 種別の決定

ユーザーに issue の種別を確認する。テンプレートが存在する場合はテンプレートに沿う。

| 種別 | 用途 |
|---|---|
| bug | バグ報告 |
| feature | 機能リクエスト |
| task | 開発タスク |
| improvement | 改善提案 |

### Step 3: 構造化された本文の生成

種別に応じた構造で本文を生成する。

#### bug テンプレート

```markdown
## 概要
（バグの概要）

## 再現手順
1. 手順 1
2. 手順 2

## 期待する動作
（期待する動作の説明）

## 実際の動作
（実際に起きている動作の説明）

## 環境
- iOS バージョン:
- デバイス:
- アプリバージョン:
```

#### feature テンプレート

```markdown
## 概要
（機能の概要）

## 背景・動機
（なぜこの機能が必要か）

## 受け入れ条件
- [ ] 条件 1
- [ ] 条件 2

## 技術的な考慮事項
（実装上の注意点）
```

#### task テンプレート

```markdown
## 概要
（タスクの概要）

## 対応内容
- [ ] 対応 1
- [ ] 対応 2

## 完了条件
- [ ] 条件 1
```

### Step 4: issue の作成

```bash
gh issue create \
  --title "<タイトル>" \
  --body "<本文>" \
  --label "<ラベル1>,<ラベル2>" \
  --assignee "<担当者>"
```

## 出力

```
## Issue 作成完了

- URL: <issue URL>
- タイトル: <タイトル>
- ラベル: <ラベル一覧>
- 担当者: <担当者>
```
